## Context

当前游戏已实现3种敌人类型（小兵/精英/BOSS）、基础AI状态机（巡逻/等待/追逐/搜索/攻击/死亡）、6种武器、波次系统、道具系统。敌人行为相对单一，缺乏策略性变化。

项目使用 Vue 3 + Three.js + Pinia 技术栈，纯前端实现，无后端依赖。敌人逻辑集中在 `apps/frontend/src/game/enemies/` 目录。

## Goals / Non-Goals

**Goals:**
- 增强BOSS战的策略性：通过阶段转换机制，让BOSS战有节奏变化
- 增加战场动态性：敌人可破坏障碍物，改变战斗格局
- 提升信息透明度：显示敌人技能冷却，让玩家有明确应对时机
- 新增2种敌人类型（自爆兵/治疗者），丰富战术选择
- 保持儿童友好的Q版卡通视觉风格

**Non-Goals:**
- 不新增外部依赖库（所有功能基于现有 Three.js 实现）
- 不修改武器系统（本次仅增强敌人行为）
- 不新增游戏模式（保持当前波次生存模式）
- 不实现敌人群体智能或编队作战（保持AI简单可理解）

## Decisions

### 1. BOSS阶段转换：基于血量阈值触发

**选择**：血量<50%时自动进入狂暴阶段，通过修改运行时属性实现，不新增敌人类型。

**理由**：避免创建新的BOSS类型配置，复用现有 `boss` 配置，通过 `phase` 字段区分阶段。

**替代方案**：
- 创建独立的 `boss-phase2` 配置 → 拒绝：会导致配置冗余，且需要修改波次配置逻辑

### 2. 障碍物破坏：扩展 CollisionDetector 支持血量

**选择**：在 `CollisionDetector` 中为每个碰撞体添加血量数据，使用 `Map<mesh.uuid, ObstacleData>` 存储。

**理由**：碰撞体数量有限（约20个），Map查找效率高。障碍物网格存储 `userData` 引用，便于渲染器访问状态。

**替代方案**：
- 创建独立的障碍物管理器 → 拒绝：增加复杂度，当前场景障碍物数量少，不需要独立管理器

### 3. 冷却显示UI：集成到现有 EnemyHealthBar

**选择**：在血条下方添加冷却进度条，复用现有的 `EnemyHealthBar` 类，不创建新组件。

**理由**：冷却条与血条强关联（同属一个敌人），放在一起逻辑清晰，且避免创建过多DOM元素。

### 4. 自爆兵实现：基于现有状态机扩展

**选择**：在 `EnemyAI.updateEnemyBehavior()` 中处理自爆逻辑，使用现有的 `attack` 状态，通过 `isCharging` 类似的机制实现预警。

**理由**：自爆兵的行为（追逐→预警→爆炸）可以映射到现有状态机，无需全新状态。

### 5. 治疗者实现：周期性扫描+治疗波

**选择**：在 `EnemyAI.update()` 中每帧检测周围友军，使用冷却时间控制治疗频率。

**理由**：治疗者不主动攻击玩家，保持巡逻状态，仅周期性触发治疗逻辑，实现简单。

## Risks / Trade-offs

- **[风险] 性能下降** → 缓解：新增粒子特效时限制数量（BOSS粒子固定16个，自爆兵预警粒子不超过8个），使用对象池思想避免频繁创建/销毁
- **[风险] 敌人AI复杂度增加** → 缓解：保持状态机结构简单，新增行为通过条件判断插入，不重构现有逻辑
- **[权衡] 障碍物破坏增加计算量** → 接受：障碍物数量有限（~20个），且破坏检测仅在敌人射击时触发
- **[权衡] 冷却UI增加DOM元素** → 接受：同屏敌人上限10个，每个新增1-2个DOM元素，总增量可接受
